Three-Schema Architecture

The Three-Schema Architecture separates user views (External), logical structure (Conceptual), and physical storage (Internal) for data independence.

External Schema (View Level)

Customized views for specific users or applications. Hides irrelevant data.

  • Example: A sales user sees only sales data.
  • Supports data independence from conceptual schema.

Conceptual Schema (Logical Level)

Defines the overall logical structure of the entire database.

  • Example: Entities like Employee, Department with relationships.
  • Independent of physical storage details.

Internal Schema (Physical Level)

Describes how data is stored physically (files, indexes, etc.).

  • Example: Data stored in B-trees or hash files on disk.
  • Optimizes storage and access efficiency.

Difference Between Schema and Instance

Schema is the blueprint/structure of the database (static), while Instance is the actual data at a point in time (dynamic).

StudentID (PK) Name (Simple Attribute) Address (Composite: Street, City) Age (Derived: Current Year - Birth Year)

Schema defines the structure (e.g., columns and types). Instance populates it with actual records.

Advantages of DBMS over Traditional File Systems

Click on cards to expand comparisons.

Data Redundancy & Inconsistency
File System: Data duplicated across files, leading to inconsistencies on updates.
DBMS: Centralized storage with integrity constraints reduces redundancy.
Data Independence
File System: Applications tightly coupled to file structures; changes require program rewrites.
DBMS: Logical/physical independence via three-schema architecture.
Concurrent Access & Security
File System: No built-in concurrency control; risk of data corruption. Limited security.
DBMS: Transaction management, locking, and user privileges ensure safe multi-user access.
Backup & Recovery
File System: Manual backups; no automatic recovery from failures.
DBMS: Automated logging, backups, and recovery mechanisms.
